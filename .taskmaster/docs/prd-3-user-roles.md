# PRD 3 — Система ролей пользователей и панель администратора

**Версия:** 1.0  
**Дата:** 2026-05-14  
**Статус:** Черновик  

---

## 1. Обзор

Данный PRD описывает добавление системы ролей в приложение Money Tracker. Главная цель — внедрить авторизацию пользователей с разделением на роли `admin` и `user`, а также создать панель администратора, где администратор может просматривать всех зарегистрированных пользователей, блокировать и удалять их.

---

## 2. Проблема

Текущее приложение:
- Не имеет аутентификации — любой посетитель видит все транзакции
- Не разграничивает доступ — нет понятия «свои» и «чужие» данные
- Не позволяет управлять пользователями — нет модерации

---

## 3. Цели

1. Внедрить регистрацию и авторизацию через Supabase Auth
2. Изолировать данные пользователей — каждый видит только свои транзакции
3. Ввести роли `admin` и `user` с разными правами доступа
4. Создать админ-панель с возможностью:
   - Просмотра списка всех зарегистрированных пользователей
   - Блокировки пользователей
   - Удаления пользователей

## 5. Пользовательские истории

### 5.1 Обычный пользователь (`user`)

- **US-1:** Как посетитель, я хочу зарегистрироваться по email и паролю, чтобы получить аккаунт
- **US-2:** Как пользователь, я хочу войти в систему, чтобы видеть свои транзакции
- **US-3:** Как пользователь, я хочу выйти из системы, чтобы завершить сеанс
- **US-4:** Как пользователь, я хочу видеть только свои транзакции, а не чужие

### 5.2 Администратор (`admin`)

- **US-5:** Как администратор, я хочу открыть админ-панель, чтобы управлять пользователями
- **US-6:** Как администратор, я хочу видеть список всех зарегистрированных пользователей (email, дата регистрации, статус)
- **US-7:** Как администратор, я хочу заблокировать пользователя, чтобы запретить ему доступ к приложению
- **US-8:** Как администратор, я хочу разблокировать пользователя, чтобы восстановить ему доступ
- **US-9:** Как администратор, я хочу удалить пользователя, чтобы полностью удалить его аккаунт и данные

## 7. Структура базы данных

### 7.1 Новая таблица: `profiles`

```sql
CREATE TABLE profiles (
  id BIGINT PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
  user_id UUID NOT NULL UNIQUE REFERENCES auth.users(id) ON DELETE CASCADE,
  email TEXT NOT NULL,
  role TEXT NOT NULL DEFAULT 'user' CHECK (role IN ('admin', 'user')),
  status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'blocked')),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Индексы
CREATE INDEX idx_profiles_user_id ON profiles(user_id);
CREATE INDEX idx_profiles_email ON profiles(email);

-- RLS
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;

-- Пользователь может читать только свой профиль
CREATE POLICY "Users can read own profile" ON profiles
  FOR SELECT USING (user_id = auth.uid());

-- Администратор может читать все профили
CREATE POLICY "Admins can read all profiles" ON profiles
  FOR SELECT USING (
    EXISTS (
      SELECT 1 FROM profiles
      WHERE user_id = auth.uid() AND role = 'admin'
    )
  );

-- Администратор может обновлять профили (смена статуса)
CREATE POLICY "Admins can update profiles" ON profiles
  FOR UPDATE USING (
    EXISTS (
      SELECT 1 FROM profiles
      WHERE user_id = auth.uid() AND role = 'admin'
    )
  );

-- Триггер: автоматическое создание профиля при регистрации
CREATE OR REPLACE FUNCTION handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO profiles (user_id, email)
  VALUES (NEW.id, NEW.email);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION handle_new_user();
```

### 7.2 Обновление таблицы `transactions`

```sql
-- Добавить колонку user_id для привязки транзакций к пользователю
ALTER TABLE transactions ADD COLUMN user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE;

-- Обновить RLS политики: пользователь видит только свои транзакции
DROP POLICY IF EXISTS "Allow read access" ON transactions;
DROP POLICY IF EXISTS "Allow insert access" ON transactions;
DROP POLICY IF EXISTS "Allow update access" ON transactions;
DROP POLICY IF EXISTS "Allow delete access" ON transactions;

CREATE POLICY "Users can read own transactions" ON transactions
  FOR SELECT USING (user_id = auth.uid());

CREATE POLICY "Users can insert own transactions" ON transactions
  FOR INSERT WITH CHECK (user_id = auth.uid());

CREATE POLICY "Users can update own transactions" ON transactions
  FOR UPDATE USING (user_id = auth.uid());

CREATE POLICY "Users can delete own transactions" ON transactions
  FOR DELETE USING (user_id = auth.uid());
```

### 7.3 Назначение первого администратора

```sql

SET role = 'admin'
```

## 12. Безопасность

| Аспект                     | Решение                                                                 |
|----------------------------|-------------------------------------------------------------------------|
| Проверка роли              | Только на сервере (Server Actions / middleware), никогда на клиенте     |
| RLS в Supabase             | Каждый SELECT/INSERT/UPDATE/DELETE проверяет user_id или роль admin     |
| Защита админ-роутов        | Middleware + Server Action двойная проверка                              |
| SQL-инъекции               | Supabase параметризованные запросы                                      |
| Само-блокировка / удаление | Запрещено на уровне UI и Server Actions                                 |
| Триггер профилей           | `SECURITY DEFINER` — работает от имени владельца БД                     |
